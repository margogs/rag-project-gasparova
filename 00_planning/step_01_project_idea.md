# Project Idea

Учебный RAG-проект на базе репозитория [MaratNotes/rag-tutorial](https://github.com/MaratNotes/rag-tutorial).
Прикладной контекст — ИИ-бот для вендингового автомата с функциональными
шот-бутылочками, который по запросу о состоянии человека подбирает подходящий витаминный шот
и объясняет его состав и пользу.

## Проблема

У стартапа «микс ит» покупатель сканирует QR и описывает состояние («нет энергии»,
«стресс», «заболеваю», «похмелье»). Боту нужно подобрать подходящие шоты и объяснить,
что в них и зачем. Без опоры на проверяемую базу состава бот либо отвечает общими
словами, либо выдумывает дозировки — это недопустимо (в т.ч. по требованиям из брифа:
запрещены заявления «лечит», «снижает болезнь»).

## Цель проекта

Сделать RAG-ассистента, который:
- по текстовому запросу о цели/состоянии находит релевантные шот-продукты в базе;
- формирует ответ **строго по данным этикеток** (состав, форма, дозировки) и показывает
  источник (`doc_id`, score, текст фрагмента);
- при отсутствии релевантных данных даёт **честный отказ**, а не выдумку.

Это «мозг» подбора для бота; вендинг, оплата, CV-камеры — вне проекта.

## Для кого проект

- Конечный сценарий: посетитель автомата «микс ит» через ИИ-бота.
- Учебный сценарий: преподаватель/проверяющий, оценивающий end-to-end RAG-pipeline.

## Какие данные используем

- **Источник:** NIH Dietary Supplement Label Database (DSLD), публичный API
  `https://api.ods.od.nih.gov/dsld/v9/`, https://dsld.od.nih.gov/
- **Что берём:** этикетки витаминных «шотов» (жидкая форма / продукты со словом *shot*):
  название, бренд, форма, ингредиенты с дозировками, заявленные на этикетке эффекты.
- **Как собираем:** скрипт `scripts/fetch_data.py` (поиск `search-filter` + карточка
  `label/{id}`) → `data/raw/datasets.json`, один продукт = одна текстовая запись.
- **Язык:** английский (как в источнике). Русскоязычный ввод — направление улучшений.
- **Масштаб:** цель «отлично» — 1000+ чанков после нарезки. Минимум для MVP — 10+ записей.
- **Лицензия:** CC0 1.0 Universal (public domain). Подробности — в `doc/DATA.md`.

> Ограничение по данным: реальный ассортимент «микс ит» немного отличается от базы DSLD.
> В MVP DSLD используется как база знаний по шотам-аналогам; это сделано в том числе с целью расширения ассортимента при необходимости
> сопоставление с каталогом «микс ит» — за рамками MVP.

## Критерий успеха

- Pipeline работает end-to-end: данные → чанки → индекс → поиск → ответ → UI.
- 3 demo-вопроса возвращают релевантные шоты с источником.
- 1 negative-вопрос (вне темы данных) корректно даёт отказ без выдумок.
- На «отлично»: 1000+ чанков, тесты green.
