# Workflow

Единый процесс работы по итерациям — для студента и проверяющего.
Опирается на `doc/tasklist.md` и `doc/conventions.md`.

## Цикл одной итерации

1. Выбрать текущую итерацию из `doc/tasklist.md`.
2. Согласовать scope: что именно делаем, какой результат и какая команда проверки.
3. Реализовать минимальным diff.
4. Прогнать команду проверки этой итерации.
5. Зафиксировать результат (коммит) и обновить статус в `tasklist.md`.
6. Перейти к следующей итерации.

## Что делать до кода

- Перечитать соответствующий раздел `tasklist.md` (задачи + проверка).
- Свериться с `conventions.md` (границы изменений, ответственность модулей, правила ответа).
- Убедиться, что итерация не тянет за собой лишний scope; если тянет — урезать до минимума.

## Как выполнять проверку

- Для каждой итерации есть своя команда из `tasklist.md`
  (`build_index.py`, `check_retrieval.py`, `check_generator.py`, `pytest`, ручной прогон UI).
- Итерация считается выполненной только если её проверка прошла.
- Поведенческие проверки (ответ/отказ) сверяем с demo- и negative-вопросами.

## Когда обновлять tasklist

- Сразу после успешной проверки итерации — менять статус ⬜ → ✅ и двигать «Текущую итерацию».
- Если scope изменился — сначала правим `tasklist.md`/`conventions.md`, потом код.

## Правила коммита и перехода дальше

- Один коммит — одна завершённая итерация (или её логический шаг).
- Сообщение коммита называет итерацию и результат (например: `iter 5: retrieval top-k + порог`).
- Переходим к следующей итерации только после green-проверки текущей.

## Запрещённые действия

- Менять несколько итераций «оптом» без проверок между ними.
- Добавлять зависимости/усложнения вне `conventions.md` (эмбеддинги, внешняя LLM в MVP).
- Править генерируемые файлы (`data/processed/`, `data/index/`) руками.
- Выдавать ответы вне найденного контекста или выдумывать состав/дозировки.
- Коммитить сырые секреты/ключи или производные артефакты, исключённые `.gitignore`.
